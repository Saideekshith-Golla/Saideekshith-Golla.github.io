System Design

Architecture overview

This setup uses a layered architecture that keeps stateful services outside of Kubernetes, improving reliability and simplifying database management.

Internal JFrog topology

This version mirrors the reference diagram style: a single system map with ingress at the edge, JFrog pods inside AKS, and managed Azure services outside the cluster boundary.

Flow summary: internal clients reach the private load balancer, nginx ingress routes requests into Artifactory, Xray integrates through Artifactory, Artifactory writes binaries to Blob Storage, Xray uses PostgreSQL, Azure DevOps deploys directly to AKS with Helm charts, and Wiz scans the cluster as part of the DevSecOps control plane.

Ingress path: Private DNS -> internal Azure Load Balancer -> nginx ingress controller -> JFrog ClusterIP services.
Compute layer: Artifactory and Xray stay in AKS as pods, while internal PostgreSQL and bundled nginx remain disabled. Wiz adds DevSecOps vulnerability scanning across the cluster.
State layer: Xray uses Azure PostgreSQL Flexible Server and Artifactory stores binaries in Azure Blob Storage outside the cluster boundary.

JFrog's internal PostgreSQL and Nginx are both disabled. All traffic is routed through an nginx ingress controller backed by an Azure internal load balancer, ensuring no public exposure.

Preparation

Prerequisites

  • An AKS cluster with at least 2 nodes (4 vCPU / 8 GB each recommended)
  • Azure Database for PostgreSQL Flexible Server provisioned
  • Azure Storage Account and container for artifact storage
  • kubectl configured against your cluster
  • Helm 3.x installed locally
  • TLS certificate and private key for your JFrog domain (.corp.example.com)
  • JFrog Helm repo added: helm repo add jfrog https://charts.jfrog.io
  • Azure AD app registration for SAML SSO (optional but recommended)
⚠
Before you begin: Store all passwords, join keys, and master keys in Azure Key Vault or a Kubernetes secret, not in plain-text values files committed to source control.
Database Setup

Step 1 - Set up PostgreSQL databases

Database Two databases are required: one for Artifactory and one for Xray. Connect to your PostgreSQL flexible server as the admin user and run the following.

Artifactory database

-- Create the Artifactory role and database
CREATE USER jfrogadmin WITH PASSWORD '<your-password>';
ALTER ROLE jfrogadmin CREATEDB;

CREATE DATABASE artifactory
  WITH OWNER=jfrogadmin
  ENCODING='UTF8';

GRANT ALL PRIVILEGES ON DATABASE artifactory TO jfrogadmin;

-- Grant schema-level permissions
GRANT USAGE, CREATE ON SCHEMA public TO jfrogadmin;
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO jfrogadmin;
GRANT ALL PRIVILEGES ON ALL SEQUENCES IN SCHEMA public TO jfrogadmin;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT ALL ON TABLES TO jfrogadmin;

Xray database

-- Create the Xray role and database
CREATE USER jfrogxray WITH PASSWORD '<your-password>';
ALTER ROLE jfrogxray CREATEDB;
GRANT jfrogxray TO postgresadmin;

CREATE DATABASE xraydb
  WITH OWNER=jfrogxray
  ENCODING='UTF8'
  lc_collate='en_US.utf8'
  lc_ctype='en_US.utf8'
  template=template0;

GRANT ALL PRIVILEGES ON DATABASE xraydb TO jfrogxray;

-- Schema ownership and access
ALTER SCHEMA public OWNER TO jfrogxray;
GRANT USAGE, CREATE ON SCHEMA public TO jfrogxray;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO jfrogxray;
ℹ
To verify permissions are in place, run SELECT has_schema_privilege('jfrogadmin', 'public', 'create');. It should return t.
Storage

Step 2 - Configure Azure Blob Storage

Storage Artifactory uses Azure Blob Storage as its binarystore backend. This keeps artifact data outside of Kubernetes and makes it independently scalable.

Create a storage account and container in Azure, then encode the credentials as a Kubernetes secret:

kubectl create secret generic artifactory-binarystore-secret \
  --from-literal=accountName=<your-storage-account> \
  --from-literal=accountKey=<your-storage-account-key> \
  -n jfrog-artifactory

In your values.yaml, configure the binarystore to point at your container:

artifactory:
  persistence:
    enabled: true
    storageClass: "managed-csi"
    size: 100Gi
  binarystore:
    type: azure-blob
    azure:
      accountName: "<your-storage-account>"
      accountKey: "<your-storage-account-key>"
      endpoint: "https://<your-storage-account>.blob.core.windows.net"
      containerName: "artifactory"
      testConnection: false
💡
Tip: Use a managed identity or workload identity on your AKS nodes to avoid managing storage account keys. Assign the Storage Blob Data Contributor role to the identity on the container.
Ingress

Step 3 - Set up the ingress controller

Infrastructure Deploy an nginx ingress controller backed by an Azure internal load balancer to ensure JFrog is not publicly exposed.

kubectl create namespace ingress-nginx

helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx

helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --set controller.service.annotations."service\.beta\.kubernetes\.io/azure-load-balancer-internal"="true" \
  --set controller.service.externalTrafficPolicy=Local \
  --set controller.replicaCount=2 \
  --set controller.nodeSelector."kubernetes\.io/os"=linux \
  --set defaultBackend.nodeSelector."kubernetes\.io/os"=linux

Create the namespaces for Artifactory and Xray:

kubectl create namespace jfrog-artifactory
kubectl create namespace jfrog-xray
Application Deploy

Step 4 - Deploy Artifactory

Artifactory With infrastructure ready, create the TLS secret and ConfigMap, then deploy using Helm.

TLS secret and certificate ConfigMap

kubectl create secret tls artifactory-tls \
  --cert=jfrog-dev.corp.example.com.cer \
  --key=jfrog-dev.corp.example.com_private_key.pem \
  -n jfrog-artifactory

kubectl create configmap artifactory-certs \
  --from-file=jfrog-dev.corp.example.com.cer \
  -n jfrog-artifactory

values.yaml

Key decisions: disable JFrog's internal PostgreSQL and Nginx, point at the external PaaS database, enable the Azure binarystore, and use ClusterIP with nginx ingress for routing.

artifactory:
  persistence:
    enabled: true
    storageClass: "managed-csi"
    size: 100Gi

  # Point at your Azure Blob binarystore (see Step 2)
  binarystore:
    type: azure-blob
    azure:
      accountName: "<your-storage-account>"
      accountKey: "<your-storage-account-key>"
      endpoint: "https://<your-storage-account>.blob.core.windows.net"
      containerName: "artifactory"
      testConnection: false

  resources:
    requests:
      cpu: "500m"
      memory: "2Gi"
    limits:
      cpu: "1"
      memory: "2Gi"

  security:
    certs:
      enabled: true
      secret: artifactory-tls

postgresql:
  enabled: false

database:
  type: postgresql
  driver: org.postgresql.Driver
  url: "jdbc:postgresql://<your-pg-server>.postgres.database.azure.com:5432/artifactory?sslmode=require"
  user: "jfrogadmin"
  password: "<your-password>"

service:
  type: ClusterIP
  ports:
    http: 8081
    https: 8443

ingress:
  enabled: true
  ingressClassName: "nginx"
  createIngressController: false
  hosts:
    - jfrog-dev.corp.example.com
  annotations:
    nginx.ingress.kubernetes.io/force-ssl-redirect: "true"
    nginx.ingress.kubernetes.io/backend-protocol: "HTTP"
    nginx.ingress.kubernetes.io/proxy-body-size: "0"
    service.beta.kubernetes.io/azure-load-balancer-internal: "true"
  tls:
    - hosts:
        - jfrog-dev.corp.example.com
      secretName: artifactory-tls

nginx:
  enabled: false

global:
  logging:
    level: INFO

Generate keys and install

Artifactory requires a masterKey and a joinKey. Generate them before installation because they cannot be changed after first boot.

# Generate keys
export MASTER_KEY=$(openssl rand -hex 32)
export JOIN_KEY=$(openssl rand -hex 32)
echo "Master: $MASTER_KEY"
echo "Join:   $JOIN_KEY"

# Install Artifactory
helm install artifactory jfrog/artifactory \
  -f values.yaml \
  --namespace jfrog-artifactory \
  --set database.type="postgresql" \
  --set artifactory.masterKey="$MASTER_KEY" \
  --set artifactory.joinKey="$JOIN_KEY"
ℹ
After deployment, verify the health endpoint with curl -s http://localhost:8082/router/api/v1/system/health after port-forwarding to the pod. You should see "status":"HEALTHY".

Upgrading (production pattern)

helm upgrade --install artifactory jfrog/artifactory \
  -f values.yaml \
  --namespace jfrog-artifactory \
  --set database.type="postgresql" \
  --set artifactory.masterKey="$MASTER_KEY" \
  --set artifactory.joinKey="$JOIN_KEY"
Identity

Step 5 - Configure SAML authentication

SSO Once Artifactory is running, configure Azure AD as a SAML identity provider. In the JFrog UI navigate to Administration → Security → Authentication → SAML and fill in the following values from your Azure AD enterprise application registration.

SAML settings
  • SAML Login URL - your Azure AD tenant's SAML2 endpoint
  • SAML Logout URL - same tenant endpoint
  • Service provider name - https://sts.windows.net/<tenant-id>/
  • Certificate - the signing certificate from your Azure AD enterprise app (Base64 PEM format)
  • Reply URL (ACS) - https://jfrog.corp.example.com/ui/api/v1/auth/saml/loginResponse/azureadintergration
💡
Two certificates: Azure AD rotates signing certificates. Download the Federation Metadata XML from your app registration and extract both the active and rollover certificates to avoid auth outages during rotation.
Security Scanning

Step 6 - Deploy Xray

Xray Xray connects to Artifactory via the join key and requires its own PostgreSQL database. The join key must exactly match the one used when deploying Artifactory.

TLS for Xray

kubectl create secret tls xray-tls \
  --cert=jfrog-dev.corp.example.com.cer \
  --key=jfrog-dev.corp.example.com_private_key.pem \
  -n jfrog-xray

kubectl create configmap xray-certs \
  --from-file=jfrog-dev.corp.example.com.cer \
  -n jfrog-xray

xray-values.yaml

xray:
  jfrogUrl: "https://jfrog-dev.corp.example.com"
  joinKey: "<same-join-key-used-for-artifactory>"

  database:
    type: postgresql
    driver: org.postgresql.Driver
    url: "jdbc:postgresql://<your-pg-server>.postgres.database.azure.com:5432/xraydb?sslmode=require"
    user: "jfrogxray"
    password: "<your-password>"

  serviceAccount:
    create: true

rbac:
  create: true

postgresql:
  enabled: false

rabbitmq:
  auth:
    password: "<rabbitmq-password>"

Install Xray

helm upgrade --install xray jfrog/xray \
  -f xray-values.yaml \
  --namespace jfrog-xray \
  --set unifiedUpgradeAllowed=true \
  --set xray.serviceAccount.create=true \
  --set xray.rbac.create=true
⚠
Master key warning: After install, Helm will warn that xray.masterKey is still at the example default. Generate a proper key with openssl rand -hex 32 and pass it via --set xray.masterKey=$KEY.

Verify Xray services are running:

kubectl get pods -n jfrog-xray
kubectl logs xray-0 -n jfrog-xray -c xray-server

Troubleshooting reference

General pod diagnostics

# List containers in a pod
kubectl get pods artifactory-0 -o jsonpath='{.spec.containers[*].name}'

# Artifactory access logs
kubectl logs -n jfrog-artifactory artifactory-0 -c access

# Xray server logs
kubectl logs xray-0 -n jfrog-xray -c xray-server

# Xray router logs
kubectl logs xray-0 -n jfrog-xray -c router

# Check running images
kubectl get pods -n jfrog-artifactory \
  -o jsonpath="{.items[*].spec.containers[*].image}"

Helm chart inspection

helm show chart jfrog/artifactory
helm show chart jfrog/xray

Restart Artifactory

# If running as a systemd service
systemctl restart artifactory

# If using the JFrog control script
$JFROG_HOME/artifactory/app/bin/artifactoryctl restart

Common PostgreSQL permission issues

If Artifactory or Xray log permission errors on startup, the most common cause is schema ownership. Run as a PostgreSQL superuser:

GRANT ALL ON SCHEMA public TO jfrogadmin;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT ALL ON TABLES TO jfrogadmin;

-- Verify
SELECT has_schema_privilege('jfrogadmin', 'public', 'usage');
SELECT has_schema_privilege('jfrogadmin', 'public', 'create');
